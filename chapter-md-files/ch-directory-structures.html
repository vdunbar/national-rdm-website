Directory Structures

Note 

The same guidelines for file naming should be applied to naming directories. 

Directories are encountered whenever file storage is needed; folders in a filing cabinet, on your desktop, on your cloud storage, etc. and act as a conduit to the actual files that contain your data. 

Conventions for directory structures and directory names will help organize your files in a way that is meaningful to both you and others. It will also allow for programmatically parsing the files if needed. 

A helpful way to start organizing folders and files is to identify natural and distinct groups of data e.g., by project, analysis type, date, etc. As well, it is important to consider how users of the files and data will access the content. 

When planning a project, it can be useful to provide a map of the directory structure being used. 

Whiteboard a Plan

Before creating folders, it is useful to white board a plan. This white board should identify directory names, directory contents, and directory access permissions. This activity has several benefits. The first is the opportunity to plan ahead and limit any re-organization of files and folders later in the project. If your project involves any kind of sensitive data to which access should be limited, this allows you to map in those access pieces. Additionally, this presents you with a descriptive map of your project’s structure, making communication of where things are and where things go a simpler task. 

Such a map may be done in several ways. Below are two potential approaches, using a subset of folders of potential relevance to a research environment. 

Graphical. Easy to navigate, potentially difficult to update. 

 

Text. Easy to update, potentially hard to read. 

|- Project 

|  |- Data [Contains restricted files and folders] 

|  |- Analysis [No restrictions] 

|  |- Funding [Restricted] 

|  |- Manuscript [No restrictions] 

Create a Template

Often, different research projects will use similar methods, allowing for templates to be created. If there is an expectation that all incoming lab members will follow a similar research life cycle approach and engage in similar research activities, such as: (1) conducting a literature review, (2) preparing a protocol, (3) collecting data, (4) presenting at conferences, etc., then a template structure pre-populated with baseline documentation, for example a readme in each folder and data dictionary in the data folder, could be prepared to help ensure adherence to best practices in file organization. Available to students in a storage location that everyone can access, this template can be copied and pasted. In more computational environments, such a template could be scripted. 

Hierarchy Depth

For the purposes of access and discoverability, directory structures can be grouped into two categories, shallow and deep. Shallow directories are characterized by minimal nesting of folders, deep directories by many layers of nesting. 

For example, in a shallow system, you may have a single directory for your data, whereas in a deep system, a directory for data would contain subdirectories, perhaps one for collected, but not yet cleaned data, one for cleaned data, one for the outputs of subsets or analyzed data etc. In turn, each of these subdirectories may include more subdirectories, such as collected, but not yet cleaned data being organized into folders by collection dates. 

It is not uncommon for people familiar with a project to prefer a shallow organizational structure. Familiar with a project would suggest that you know what files exist, and can generally locate them easily amongst a long list of other files. When unfamiliar with a project, a deep structure may be advantageous as descriptive and meaningful folder names can guide a user through the folder contents. In this way, deep structures offer more metadata. 

In some setups, specific directories will be shallow, and others deep; it is not necessary to be consistent across all directories associated with a specific project. But you should consider the impacts of the directory structure on other potential users or viewers of their contents. 

It is also possible that the depth and scope of a directory structure will be largely informed by the size and scope of the information it is attempting to organize; smaller projects often require less complex systems than large projects. 

Common Directories

All projects should be contained in a unique root directory; the root directory here refers to the highest level directory associated with the project. Subdirectories then often benefit from thinking about the various activities, or functions, associated with a research project. Frequently these activities include: 

  • Conducting a literature review 

  • Writing a protocol or research plan 

  • Collecting data 

  • Analyzing data 

  • Communicating findings 

A directory structure that might map to this would be (colour coded to hierarchical level) 

|- top-level-project-folder 

|  |- literature-review 

|  |  |- pdfs 

|  |  |- writing 

|  |- data-collection 

|  |  |- data-collection-tools 

|  |  |- unprocessed-data 

|  |- analysis 

|  |  |- scripts 

|  |  |- processed-data 

|  |- manuscripts-conferences 

|  |  |- manuscripts 

|  |  |- conferences 

An activity, or functional, approach like this has several benefits. One of the key ones for research data management is that it encourages a separation between collected data and data used for analysis. That is, unprocessed data is kept separate from any analyses on these data so there is always an original copy of the data that can be referenced. 

Note 

Increasingly, the idea that we might have a directory structure that contains all inputs and outputs related to a research project is less and less tenable, especially when using collaborative, cloud-based platforms and tools or specialized applications that handle aspects of file management for the user. Examples include online editing platforms, citation managers, etc. However, the conceptual framework from the standpoint of research data management remains of vital importance. It is for this reason that a map articulating the structural components of a research project and where these reside, whether this is articulating a directory structure or instead a network of disconnected platforms, tools, and applications is crucial to research data management.
